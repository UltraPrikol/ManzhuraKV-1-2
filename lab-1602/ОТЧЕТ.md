#### **1.2. Интерфейс Airtable (основные элементы)**

| Элемент | Описание |
|---------|----------|
| **Base** | База данных, содержит несколько таблиц (Grid views) |
| **Table** | Таблица, аналогична листу в Excel или таблице SQL |
| **Field** | Колонка с определённым типом данных (текст, число, дата, ссылка) |
| **Record** | Строка в таблице (запись) |
| **View** | Представление (фильтрация, сортировка, группировка) |
| **Form** | Форма для ввода данных, создаётся автоматически |
| **Extension** | Расширения (графики, календари, диаграммы) |

### Контрольные вопросы
- Какие ограничения low-code подхода вы обнаружили при работе с Airtable?
    Сложность в реализации кастомной бизнес-логики, которая выходит за рамки встроенных формул. Невозможность тонкой настройки пользовательского интерфейса (UI) и ограниченные возможности для контроля производительности при сложных запросах.
- Какие типы полей или валидаций вам не хватило? Что пришлось бы делать через код?
    Не хватает валидации по регулярным выражениям (Regex) встроенными средствами, сложных составных проверок на стороне клиента перед отправкой формы и транзакционных операций (например, списывание остатков со склада строго в момент подтверждения оплаты). Это пришлось бы делать кодом через API.
- В каких сценариях low-code платформа эффективнее традиционной разработки?
    При создании MVP (минимально жизнеспособного продукта), автоматизации внутренних процессов небольших команд, создании CRM-систем и админ-панелей, где критична скорость запуска (Time-to-Market), а не уникальность интерфейса.
- Что такое "vendor lock-in" и как он проявляется в Airtable?
    Это привязка к поставщику услуг. В Airtable она проявляется в том, что вся бизнес-логика (формулы, связи, автоматизации) существует только внутри их экосистемы. При миграции можно выгрузить только сырые данные (CSV/JSON), но всю архитектуру приложения придется писать с нуля.
- Как бы вы масштабировали это приложение при росте до 1 млн заказов в день?
    Airtable не справится с такой нагрузкой (лимит записей на базу — 100-250 тыс. в зависимости от тарифа). Потребуется миграция на традиционный стек: реляционная СУБД (PostgreSQL/MySQL) для хранения данных и backend-сервер (например, на Python/Node.js) для обработки логики, кэширование через Redis.
- Какие российские аналоги Airtable вы знаете? (Cetera, Appmaster.io, Puzl)
    Cetera, Appmaster.io, Puzl, Yandex Tracker (в качестве системы управления процессами).
