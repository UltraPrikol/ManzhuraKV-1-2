### Контрольные вопросы
- Какие преимущества даёт многостадийная сборка в Dockerfile?
  Минимальный размер образа: В финальный образ не попадают лишние тяжелые инструменты (компиляторы вроде gcc, заголовочные файлы libpq-dev, кэш менеджера пакетов). Там остается только рантайм и сам работающий код.
  Безопасность: Чем меньше лишнего софта в контейнере, тем меньше потенциальных уязвимостей (меньше attack surface), которые злоумышленник может использовать при взломе.
- Почему для фронтенда выбран Vercel, а для бэкенда Yandex Cloud?
  Vercel (Фронтенд): Заточен под статику и SPA (React, Vue, Next.js). Он автоматически берет код из GitHub, раздает его через глобальный CDN (мгновенная загрузка в любой точке мира) и абсолютно бесплатен на базовом тарифе. При этом собирать Docker-образы для фронта вручную там не нужно.
  Yandex Cloud (Бэкенд): Предоставляет полноценную облачную инфраструктуру внутри РФ с минимальным пингом. Позволяет развернуть полноценную реляционную базу данных (Managed PostgreSQL) в изолированной сети и контейнеризированный бэкенд (FastAPI), обеспечивая надежную работу с данными и соблюдение требований по локализации персональных данных.
- Как решается проблема CORS при обращении фронтенда к API в разных доменах?
  Проблема решается на стороне бэкенда с помощью CORS-middleware.
  В коде FastAPI (main.py) подключается модуль CORSMiddleware. Бэкенд проверяет заголовок Origin входящего запроса от фронтенда (например, https://manzhura-kv-1-2.vercel.app). Если этот адрес совпадает со списком разрешенных в переменной CORS_ORIGINS, бэкенд добавляет в свой ответ заголовок Access-Control-Allow-Origin. Браузер видит этот заголовок и разрешает фронтенду прочитать ответ от API.
- Какие переменные окружения необходимо хранить в секрете, а какие могут быть публичными?
  Секретные (Только бэкенд, скрыты в ТГ/Облаке): Пароли, токены, приватные ключи и полные строки подключения к БД (например, DATABASE_URL, содержащая пароль пользователя базы bookuser:SecurePassword123). Их утечка ведет к компрометации данных.
  Публичные (Можно в код или открытый конфиг): Настройки окружения (PYTHONUNBUFFERED=1), порты приложения (PORT=8080), адреса разрешенных фронтендов (CORS_ORIGINS) или базовый URL публичного API, который и так виден в сетевых запросах браузера.
- Что такое "холодный старт" в Cloud Run и как он влияет на производительность?
  Что это: Это задержка при первом обращении к Serverless-контейнеру (в Yandex Cloud / Cloud Run), когда облаку приходится «с нуля» выделять виртуальную машину, скачивать туда Docker-образ и запускать твое приложение FastAPI, потому что до этого контейнер «спал» для экономии ресурсов.
  Влияние: Первый пользователь после простоя (или при резком скачке трафика) получит сильную задержку ответа (от 2 до 10+ секунд) вместо привычных миллисекунд. На последующие запросы контейнер ответит мгновенно, так как перейдет в «горячее» (активное) состояние.
