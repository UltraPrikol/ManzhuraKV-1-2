## Теоретическая часть
**Rust как системный функциональный язык:**
Rust сочетает производительность системных языков с безопасностью и выразительностью функциональных языков. Гарантии безопасности памяти проверяются на этапе компиляции.

**Основные концепции ФП в Rust:**
*   **Иммутабельность по умолчанию** - переменные неизменяемы без ключевого слова `mut`
*   **Система владения** - гарантии безопасности памяти без сборщика мусора
*   **Трейты** - аналоги type classes из Haskell
*   **Алгебраические типы данных** - enum с данными
*   **Сопоставление с образцом** - exhaustive pattern matching

**Ключевые особенности:**
*   **Нулевая стоимость абстракций** - функциональные конструкции не добавляют runtime накладных расходов
*   **Отсутствие исключений** - обработка ошибок через Result и Option
*   **Ленивые итераторы** - эффективные цепочки преобразований

struct Product {
    name: String,
    price: f64,
    available: bool,
}

#[derive(Debug)]
struct Order {
    id: u32,
    amount: f64,
}

#[derive(Debug)]
enum OrderError {
    InvalidAmount(u32),
}

// --- Задание 1: Анализ продуктов ---
fn analyze_products(products: &[Product]) -> (f64, usize, Vec<&Product>) {
    let count = products.len();
    
    // Средняя цена (если список пуст, вернем 0.0)
    let avg_price = if count == 0 {
        0.0
    } else {
        products.iter().map(|p| p.price).sum::<f64>() / count as f64
    };

    // Количество доступных
    let available_count = products.iter().filter(|p| p.available).count();

    // Список дорогих продуктов (> 100)
    let expensive_products: Vec<&Product> = products
        .iter()
        .filter(|p| p.price > 100.0)
        .collect();

    (avg_price, available_count, expensive_products)
}

// --- Задание 2: Валидация цепочки заказов ---
fn validate_orders(orders: &[Order]) -> Result<Vec<&Order>, OrderError> {
    // В Rust метод collect может превратить Iterator<Result<T, E>> в Result<Vec<T>, E>
    // Он вернет первую встреченную ошибку или вектор всех успешных элементов.
    orders.iter()
        .map(|o| {
            if o.amount > 0.0 {
                Ok(o)
            } else {
                Err(OrderError::InvalidAmount(o.id))
            }
        })
        .collect()
}

struct Fibonacci {
    current: u64,
    next: u64,
}

impl Fibonacci {
    fn new() -> Self {
        Fibonacci { current: 0, next: 1 }
    }
}

impl Iterator for Fibonacci {
    type Item = u64;
    
    fn next(&mut self) -> Option<Self::Item> {
        let current_val = self.current;
        
        // Математическая логика: F(n) = F(n-1) + F(n-2)
        let next_val = self.current.checked_add(self.next);
        
        match next_val {
            Some(sum) => {
                self.current = self.next;
                self.next = sum;
                Some(current_val)
            }
            None => None, // Защита от переполнения u64
        }
    }
}

**Контрольные вопросы**
1.  Как система владения Rust обеспечивает безопасность памяти?
Система владения базируется на трех правилах: у каждого значения есть один владелец, при выходе владельца из области видимости значение удаляется, 
передача значения другому владельцу делает старого владельца недействительным. 
Это позволяет Rust управлять памятью без сборщика мусора и предотвращать ошибки вроде двойного освобождения или обращения к висячим ссылкам.

2.  В чем разница между `&T` и `&mut T`?
&T является разделяемой (неизменяемой) ссылкой. Таких ссылок может быть сколько угодно одновременно. 
&mut T является исключительной (изменяемой) ссылкой. Она может существовать только в единственном экземпляре для данных в конкретный момент времени. 
Правила Rust запрещают совмещать изменяемую ссылку с любыми другими ссылками на те же данные.

3.  Как работают ленивые итераторы в Rust?
Итераторы в Rust не выполняют вычислений сразу после создания. Они только описывают последовательность действий. 
Реальная работа начинается только в момент вызова "потребляющего" метода (например, collect, sum или fold). 

Это позволяет оптимизировать производительность и избегать лишних выделений памяти под промежуточные результаты.
4.  Какие преимущества дают алгебраические типы данных и pattern matching?
Алгебраические типы (Enum) позволяют объединять разные структуры данных под одним типом. 
Pattern matching (сопоставление с образцом) заставляет программиста на этапе компиляции обработать все возможные варианты развития событий. 
Это исключает логические ошибки, когда какой-то редкий случай был забыт в коде.

5.  Почему Rust предпочитает Result и Option вместо исключений?
Эти типы делают ошибки явными значениями. Вместо того чтобы скрыто прерывать поток выполнения программы, функция сообщает в своей сигнатуре о возможности неудачи. 
Это заставляет разработчика обрабатывать ошибки сразу, что повышает надежность системы и делает логику работы программы полностью прозрачной.
